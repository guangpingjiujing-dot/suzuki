# セマンティックレイヤーとは

## ひとことで

**セマンティックレイヤー**は、分析で使う指標（メトリクス）や切り口（ディメンション）などの「意味」を **YAML などで一度定義し、ダッシュボードや SQL クライアントなどから一貫した形で使えるようにする層**のことです。

同じ「売上」を、ツール A では `SUM(amount)`、ツール B では別名の列で別ロジックが走る、といった **定義のブレ**を減らすのが目的のひとつです。

## dbt の文脈で何が増えるか

dbt では主に次のようなブロックをプロジェクトに書きます。


| 概念                 | 役割のイメージ                                                               |
| ------------------ | --------------------------------------------------------------------- |
| **Semantic Model** | どの dbt モデルを粒度の土台にするか、**エンティティ**（例: 顧客 ID）や **ディメンション**（例: 地域、日付）を宣言する |
| **Metric**         | その上で「**何を数える・足す・割るか**」などの指標を名前付きで定義する                                 |
| **Saved Query**    | メトリクスとグループ化の組み合わせを名前で保存し、再利用やエクスポートの単位にする                             |


これらは通常の `models/*.sql` のモデル定義と **別の YAML** に並んでいます（jaffle-shop では `models/marts/customers.yml` などに `semantic_models` / `metrics` / `saved_queries` が入っています）。

## ローカルで「挙動を確かめる」とは

- **dbt Core** … プロジェクトをパースしたり、モデルをビルドしたりする（`dbt parse`, `dbt run` など）。
- **MetricFlow（`mf` コマンド）** … 上記の YAML 定義を読み、**メトリクス一覧の表示**や **実際にデータウェアハウスへクエリを投げて結果を返す**などができる（dbt Cloud の `dbt sl` とは別ルートだが、ローカル検証には向いている）。

このドキュメントでは **BigQuery 上の jaffle-shop** を例に、パースから `mf query` まで一通り触る手順をまとめます。

対象フォルダ: `suzuki/semantic_layer/jaffle-shop`  
前提: **BigQuery** にデータを載せられること（このリポジトリの `profiles.yml` は BigQuery / OAuth を想定）。

## 1. dbt パッケージの取得

`packages.yml` にあるパッケージ（例: `dbt_utils`）を取り込みます。

```powershell
uv run dbt deps
```

## 2. シードとモデルのビルド

セマンティック定義の土台になるテーブルを BigQuery に作ります。

```powershell
uv run dbt seed
uv run dbt run
```

（初回は BigQuery のジョブが走るため、数分かかることがあります。）

## 3. セマンティック定義の反映（パース）

メトリクスやセマンティックモデルを変更したあとでも、**最低一度は**パースしてマニフェストを更新するのが安全です。

```powershell
uv run dbt parse
```

エラーや重大な警告がなければ、ここまでで「定義として整合している」状態です。

## 4. MetricFlow でメタデータを見る

メトリクス名と、クエリ時に使えるディメンションの一覧を確認します。

```powershell
uv run mf list metrics
uv run mf list entities --metrics orders 
uv run mf list dimensions --metrics orders
uv run mf list dimension-values --metrics orders --dimension customer__customer_name
uv run mf list saved-queries
```

`orders` のように実在するメトリクス名に置き換えて試してください。一覧は `mf list metrics` の出力に出ます。

## 5. MetricFlow で実データを問い合わせる

単純な例（行数を制限して結果だけ見る）:

```powershell
uv run mf query --metrics orders --limit 10
```

**どんな SQL が飛んだか**まで見たい場合は `--explain` を付けます。

```powershell
uv run mf query --metrics orders --limit 10 --explain
```

ディメンションでグループ化する例（利用可能な名前は `mf list metrics` の行に依存します）:

```powershell
uv run mf query --metrics orders --group-by metric_time --limit 20
```

プロジェクトに定義されている **保存クエリ**を使う例（`models/marts/customers.yml` の `saved_queries` に `customer_order_metrics` があります）:

```powershell
uv run mf query --saved-query customer_order_metrics --limit 20
```

## 6. 設定の検証だけ行う（任意）

ウェアハウスに重いクエリを投げず、YAML の整合性をチェックする場合:

```powershell
uv run mf validate-configs
```

## つまずきやすいポイント


| 現象                        | 確認すること                                                                                                                   |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| BigQuery 認証エラー            | `gcloud auth application-default login`、プロジェクト課金・API 有効化                                                                 |
| `profiles.yml` が見つからない    | `DBT_PROFILES_DIR` が `jaffle-shop` を指しているか                                                                               |
| `mf` が古い定義のまま             | メトリクス変更後に `uv run dbt parse` を再実行                                                                                        |
| Google の「quota project」警告 | 開発には支障がないことが多いが、気になる場合は [ADC のトラブルシュート](https://cloud.google.com/docs/authentication/adc-troubleshooting/user-creds) を参照 |
